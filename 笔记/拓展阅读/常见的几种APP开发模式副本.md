# 常见的几种APP开发模式

## ReactNative

- 与传统跨平台开发不同,它的目标不是一次编写到处执行,而是一次学习多处开发(一次学习跨平台开发)-不同平台上编写基于React的代码

- 其中Android版本和iOS版本代码是有差别的,但是由于都是js驱动开发,相同语法的,所以会编写iOS的同时也会编写Android

- 相比传统跨平台大大提升了性能-既拥有Native的用户体验、又保留React的开发效率

- 原理仍然是依赖原生暴露的接口-通过JSBridge技术,所以仍然会存在原生代码
参考1:http://blog.cnbang.net/tech/2698/  参考2: http://reactnative.cn/

- reactNative逻辑是用js写的-react语法，但是UI全部是原生的，利用Bridge技术,将js渲染成android和ios代码执行了

- 和NaviveJS的区别: NativeJS说到底view层还是由dom驱动的,只不过是调用了原生暴露的API
但是RN的view层是由虚拟dom驱动的

- ReactJs和React Native的原理是相同的，都是由js实现的虚拟dom来驱动界面view层渲染。
只不过ReactJs是驱动html dom渲染； React Native是驱动android/ios原生组件渲染。
其实在React Native推出之前，就已经存在这种使用js驱动app原生组件的技术了，比如Native Script。

## Weex

- Weex最底层的原理是和React-Native相同的，就是将JS代码渲染成原生组件

- 只不过在业务代码层面，Weex和React-Native有差别
    - 最新版Weex是基于Vue的语法，然后写出来的就是一些特定的H5语法，然后底层引擎再分别根据web和原生，分别将代码解析成html dom或者原生组件，所以感觉起来跟写web插件一样。
    - 而React-Native则是直接通过JS写原生代码，所以对外表现层上，weex只要遵循一定规则就可以一套代码完全跨平台，React-Native则针对不同平台写不同代码，虽然大部分代码可以复用
    - 维护层面上，Weex主要由阿里人员维护，社区方面目前还有点不足，而React-Native目前社区已经比较成熟，期待未来两者各自的发展

## Native APP

- 传统的原生开发，分别用各自平台的开发语言

- 优点是性能最高

- 缺点是开发维护成本高,复用性差,iOS版本更新也成问题

## Web APP

- 这里一般只指SPA形式的,因为MPA形式的都不能算是应用,只能算网站
因为SPA形式的往往会模拟APP做一些交互功能
而且严格来说,这并不能属于应用

### 优点:
1.用户不需要安装,不会占用手机内存,并且通过浏览器即可访问,迭代最快
2.开发起来调试也是最为方便的,直接可以通过浏览器调试

### 缺点:
1.用户体验不够
2.由于实质还是网站形式,所以必须联网

## Hybrid App

### ejs

- 介于原生APP与webAPP之间,但是由原生开发主导,部分展示页面用webview形式加载h5页面显示

- 原生通过暴露一些JSAPI给h5页面调用-JSBridge,
这样就实现了原生与h5的结合,提高开发效率

- 什么是JSBridge:实现java（ios为oc）和js的互相调用的桥梁。
替代了WebView的自带的JavascriptInterface的接口，使得我们的开发更加灵活和安全。
因为普通的webview 接口在某些存在安全隐患

- 主要的交互就是通过原生定义的一些暴露API规范

#### 优点: 
1. 性能与原生相比基本没有损耗(因为只是一些简单页面用h5,复杂实现都是原生),性能高于Hybird模式应用
2. h5开发的页面能够多平台复用,并且这部分功能能由服务器快速更新迭代
3. 与原生相比,提高了开发效率
4. h5部分功能复用方便,比如突然某一个模块要做一个h5版本或微信版本,直接可以用现成页面
 

#### 缺点:
1. 相比Hybird模式,复用度不够,开发速度没有这么快,版本迭代也没有这么自由
2. 需要对应平台原生人员的配合,或者说开发人员得具备原生开发知识,初学者无法以一己之力完成开发

### Html5+

#### 严格的说,这种模式也属于NativeJS形式
但区别就是一般这种模式都会有一个固定的第三方SDK(如Dcoud,AppCan提供的) 
这些第三方SDK会提供统一的调用API,然后针对不同平台自己实现功能(开发者不可见)
一般这些第三方SDK会提供完整成熟的API,并且会有入口页面功能,里面自带会实现不同页面的切换逻辑
这样,开发者只需要用js 来写逻辑和调用API,然后通过云端打包等功能,就可马上得到完整的应用

#### 优点: 
1. 开发便捷,一套代码,多个平台适用，开发成本低,
2. 针对新手友好,初学者甚至不需要了解原生的实现,只要调用API就能完整一个完整应用
3. 版本迭代较为自由,可以自行代码操控
4. 相对而言,比起纯web应用,性能和体验要好很多
5. 功能复用方便,同NativeJS形式

#### 缺点:
1. 相比起原生,性能有所损耗,部分交互性复杂的页面损耗会更严重
2. 由于实际上使用的是第三方SDK,所以如果第三方SDK出问题基本上无法解决-存在比较重的依赖现象
